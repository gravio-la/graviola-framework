# Pipeline Stages

The metadata extraction framework transforms data through four distinct stages. Each stage performs specific transformations and passes data to the next stage in a clean, composable pipeline.

## Stage Overview

## Stage 1: Schema Normalization

**Package**: `@graviola/edb-graph-traversal`  
**Function**: `normalizeSchema(schema, filterOptions)`

### Purpose

Convert a JSON Schema with `$ref` references into a flat, resolved schema ready for SPARQL generation.

### Input

```typescript
{
  type: "object",
  properties: {
    name: { type: "string" },
    friends: {
      type: "array",
      items: { $ref: "#/$defs/Person" }  // Reference!
    }
  },
  $defs: {
    Person: {
      type: "object",
      properties: {
        name: { type: "string" },
        age: { type: "number" }
      }
    }
  }
}
```

### Transformation Process

<Mermaid chart={`graph LR
    A[Schema<br/>with $refs] --> B{Has $refs?}
    B -->|Yes| C[Resolve Reference]
    C --> D{Max Depth?}
    D -->|No| C
    D -->|Yes| E[Insert Stub]
    B -->|No| F[Apply Filters]
    E --> F
    F --> G[Extract Metadata]
    G --> H[Add Pagination]
    H --> I[Normalized<br/>Schema]
    
    style A fill:#e1f5ff
    style I fill:#fff4e1
`} />

### Transformations

#### 1. Reference Resolution

Recursively resolves all `$ref` pointers up to a configurable depth (default: 4 levels).

```typescript
// Before
{ $ref: "#/$defs/Person" }

// After (resolved inline)
{
  type: "object",
  properties: {
    name: { type: "string" },
    age: { type: "number" }
  }
}
```

#### 2. Filter Application

Applies Prisma-style filters to determine which properties to include:

```typescript
// Filter options
{
  select: { name: true },      // Only include 'name'
  omit: ["internalId"],          // Exclude 'internalId'
  include: {                      // Include relationship with pagination
    friends: {
      take: 10,
      orderBy: { name: 'asc' }
    }
  }
}
```

#### 3. Relationship Identification

Marks properties as relationships based on:
- Presence of `@id` property in object schema
- `x-relationship: true` hint in schema

```typescript
_propertyMetadata: {
  friends: {
    isRelationship: true,  // Marked as relationship
    isArray: true,
    itemType: "object"
  }
}
```

#### 4. Pagination Metadata

Attaches pagination configuration to array properties:

```typescript
{
  type: "array",
  items: { /* ... */ },
  "x-pagination": {
    take: 10,
    skip: 0,
    orderBy: { name: 'asc' }
  }
}
```

### Output

```typescript
{
  type: "object",
  properties: {
    name: { type: "string" },
    friends: {
      type: "array",
      items: {
        type: "object",
        properties: {
          name: { type: "string" },
          age: { type: "number" }
        }
      },
      "x-pagination": {
        take: 10,
        orderBy: { name: 'asc' }
      }
    }
  },
  _normalized: true,
  _propertyMetadata: { /* ... */ }
}
```

## Stage 2: SPARQL Generation

**Package**: `@graviola/sparql-schema`  
**Function**: `normalizedSchema2construct(iri, normalizedSchema, options)`

### Purpose

Convert a normalized schema into SPARQL CONSTRUCT and WHERE patterns, including SUBSELECTs for paginated arrays.

### Input

Normalized schema from Stage 1 + subject IRI:

```typescript
normalizedSchema2construct(
  "http://example.com/person/1",  // Subject IRI
  normalizedSchema,                // From Stage 1
  { prefixMap: { foaf: "..." } }  // Optional prefixes
)
```

### Transformation Process

<Mermaid chart={`graph TD
    A[Normalized Schema] --> B[Walk Properties]
    B --> C{Property Type?}
    C -->|Literal| D[Generate Triple Pattern]
    C -->|Object| E[Generate + Recurse]
    C -->|Array| F{Has Pagination?}
    F -->|No| D
    F -->|Yes| G[Create SUBSELECT]
    G --> H[Add ORDER BY]
    H --> I[Add LIMIT/OFFSET]
    I --> J[Combine Patterns]
    D --> J
    E --> J
    J --> K[CONSTRUCT + WHERE]
    
    style A fill:#fff4e1
    style K fill:#ffe1f5
`} />

### Generated Patterns

#### CONSTRUCT Clause

Defines which triples to extract from the graph:

```sparql
CONSTRUCT {
  <http://example.com/person/1> :name ?name .
  <http://example.com/person/1> :friends ?friends_0 .
  ?friends_0 :name ?name_1 .
  ?friends_0 :age ?age_1 .
}
```

#### WHERE Clause (Simple)

Basic pattern matching:

```sparql
WHERE {
  <http://example.com/person/1> :name ?name .
  OPTIONAL { <http://example.com/person/1> :friends ?friends_0 }
}
```

#### WHERE Clause (with SUBSELECT)

For paginated arrays with ORDER BY:

```sparql
WHERE {
  <http://example.com/person/1> :name ?name .
  
  OPTIONAL { {
    SELECT ?friends_0 WHERE {
      <http://example.com/person/1> :friends ?friends_0 .
      OPTIONAL { ?friends_0 :name ?name }
    }
    ORDER BY ASC(?name)
    LIMIT 10
  } }
  
  OPTIONAL { ?friends_0 :name ?name_1 }
  OPTIONAL { ?friends_0 :age ?age_1 }
}
```

### Output

```typescript
{
  constructPatterns: SparqlTemplateResult[],  // CONSTRUCT patterns
  wherePatterns: SparqlTemplateResult[],      // WHERE patterns
  paginationMetadata: Map<string, {           // Pagination tracking
    take: 10,
    orderBy: { name: 'asc' },
    source: "query"  // Marked as query-stage pagination
  }>
}
```

## Stage 3: Query Execution

**External Package**: Your SPARQL client (e.g., `sparql-http-client`)

### Purpose

Execute the generated SPARQL CONSTRUCT query against an RDF triple store and return results as an in-memory RDF dataset.

### Process

```typescript
// Build complete query with PREFIX declarations
const query = buildQuery(constructPatterns, wherePatterns, prefixes);

// Execute against triple store
const stream = await sparqlClient.query.construct(query);

// Convert to RDF/JS dataset
const dataset = await datasetExt.fromStream(
  datasetFactory.dataset(),
  stream
);
```

### Input

Complete SPARQL query string:

```sparql
PREFIX : <http://example.com/>
PREFIX foaf: <http://xmlns.com/foaf/0.1/>

CONSTRUCT {
  <http://example.com/person/1> :name ?name .
  <http://example.com/person/1> :friends ?friends_0 .
  ?friends_0 :name ?name_1 .
}
WHERE {
  <http://example.com/person/1> :name ?name .
  OPTIONAL { { SELECT ?friends_0 ... } }
  OPTIONAL { ?friends_0 :name ?name_1 }
}
```

### Output

RDF/JS Dataset containing matching triples:

```typescript
{
  // RDF quads in memory
  <http://example.com/person/1> :name "John" .
  <http://example.com/person/1> :friends _:b1 .
  _:b1 :name "Jane" .
  _:b1 :age 30 .
}
```

## Stage 4: Graph Extraction

**Package**: `@graviola/edb-graph-traversal`  
**Function**: `extractFromGraph(iri, dataset, normalizedSchema, options)`

### Purpose

Walk the RDF dataset using the normalized schema to extract a type-safe JavaScript object.

### Input

- Subject IRI: `"http://example.com/person/1"`
- RDF Dataset from Stage 3
- Normalized Schema from Stage 1
- Base IRI for property expansion

### Transformation Process

<Mermaid chart={`graph TD
    A[Start at Subject IRI] --> B[Get Clownface Pointer]
    B --> C[Walk Schema Properties]
    C --> D{Property Type?}
    D -->|Literal| E[Extract Literal Value]
    E --> F[Apply Type Coercion]
    F --> L
    D -->|NamedNode| G[Follow to New Node]
    G --> H[Recurse if depth allows]
    H --> L
    D -->|BlankNode| I[Follow Anonymous Node]
    I --> H
    D -->|Array| J{Has Pagination?}
    J -->|No| K[Extract All Items]
    J -->|Query Stage| K
    J -->|Extraction Stage| M[Apply Pagination]
    M --> K
    K --> L[Combine Properties]
    L --> N[Return Object]
    
    style A fill:#e1ffe1
    style N fill:#f0e1ff
`} />

### Extraction Logic

#### 1. Literal Extraction

```typescript
// RDF: <person> :name "John"
// Schema: { type: "string" }
// Result: "John"

// RDF: <person> :age "30"^^xsd:integer
// Schema: { type: "number" }
// Result: 30  (type coerced)
```

#### 2. Object Extraction (NamedNode)

```typescript
// RDF: <person> :employer <http://example.com/company/1>
// Schema: { type: "object", properties: { name: { type: "string" } } }
// Result: { "@id": "http://example.com/company/1", name: "Acme Corp" }
```

#### 3. Object Extraction (BlankNode)

```typescript
// RDF: <person> :address _:b1 . _:b1 :city "NYC"
// Schema: { type: "object", properties: { city: { type: "string" } } }
// Result: { city: "NYC" }  // No @id for blank nodes
```

#### 4. Array Extraction with Pagination

```typescript
// x-pagination source check:
if (paginationMeta?.source === "query") {
  // Already paginated at query stage - use all items
  return items;
} else if (paginationMeta) {
  // Apply pagination during extraction
  return items.slice(skip, skip + take);
}
```

### Output

Type-safe JavaScript object:

```typescript
{
  "@id": "http://example.com/person/1",
  name: "John",
  friends: [
    { name: "Jane", age: 30 },
    { name: "Bob", age: 25 },
    // ... 10 items total (paginated)
  ]
}
```

## Performance Considerations

### Stage 1: Normalization

- **Cached**: Normalized schemas can be reused across multiple queries
- **Cost**: O(depth × properties) for ref resolution
- **Optimization**: Set appropriate maxRecursion to avoid deep nesting

### Stage 2: SPARQL Generation

- **Cached**: SPARQL patterns can be cached if schema doesn't change
- **Cost**: O(properties) to walk schema
- **Optimization**: Use query-stage pagination for large arrays

### Stage 3: Query Execution

- **Network Bound**: Depends on triple store performance
- **Cost**: Varies by query complexity and data size
- **Optimization**: Use SUBSELECT to limit data transfer

### Stage 4: Extraction

- **Memory Bound**: Works on in-memory dataset
- **Cost**: O(triples) to walk graph
- **Optimization**: Skip extraction-stage pagination if done at query stage

# Key Concepts

Understanding these core concepts will help you effectively use the metadata extraction framework.

---

## Prisma-Style Filters

The framework uses a filtering API inspired by [Prisma ORM](https://www.prisma.io/), making it familiar and intuitive for modern TypeScript developers.

### Select Pattern

**Purpose**: Pick specific fields to include in the result.

```typescript
{
  select: {
    name: true,      // Include name
    email: true,     // Include email
    // password is excluded (not selected)
  }
}
```

**When to use**: When you know exactly which fields you need and want to exclude everything else.

**Result**:
```typescript
{ name: "John", email: "john@example.com" }
// password, address, etc. are not included
```

### Include Pattern

**Purpose**: Explicitly include relationships with optional pagination.

```typescript
{
  include: {
    friends: true,                    // Include all friends
    posts: {                          // Include posts with pagination
      take: 10,
      skip: 0,
      orderBy: { createdAt: 'desc' }
    }
  }
}
```

**When to use**: When you want to control which relationships are fetched and how they're paginated.

**Result**:
```typescript
{
  name: "John",
  friends: [{ name: "Jane" }, { name: "Bob" }],  // All friends
  posts: [/* 10 most recent posts */]            // Paginated
}
```

### Omit Pattern

**Purpose**: Exclude specific fields from the result.

```typescript
{
  omit: ["password", "internalId", "createdAt"]
}
```

**When to use**: When you want most fields but need to exclude sensitive or unnecessary data.

**Result**:
```typescript
{
  name: "John",
  email: "john@example.com",
  // password, internalId, createdAt are excluded
}
```

### Combining Filters

You can combine different filter patterns:

```typescript
{
  select: { name: true, email: true },  // Only these fields
  omit: ["email"],                       // But exclude email
  // Result: { name: "John" }
}
```

**Precedence**:
- `omit` is checked first (always excludes)
- `select` is checked next (if specified, only selected fields)
- `include` determines which relationships to fetch

## Pagination Options

### Basic Pagination

```typescript
{
  take: 10,    // Return 10 items (LIMIT)
  skip: 20,    // Skip first 20 items (OFFSET)
}
```

**SQL Equivalent**:
```sql
LIMIT 10 OFFSET 20
```

### Pagination with Ordering

```typescript
{
  take: 10,
  skip: 0,
  orderBy: { name: 'asc' }  // Sort by name ascending
}
```

**SPARQL Equivalent**:
```sparql
ORDER BY ASC(?name)
LIMIT 10
```

### Multiple Sort Criteria

```typescript
{
  orderBy: [
    { age: 'desc' },   // Primary sort: age descending
    { name: 'asc' }    // Secondary sort: name ascending
  ]
}
```

**SPARQL Equivalent**:
```sparql
ORDER BY DESC(?age) ASC(?name)
```

### Why orderBy Matters

See the [Blank Nodes vs Named Nodes](#blank-nodes-vs-named-nodes) section below.

## Blank Nodes vs Named Nodes

In RDF, nodes can be either **Named Nodes** (with URIs) or **Blank Nodes** (anonymous).

### Named Nodes

**Definition**: Nodes with a URI identifier.

```turtle
<http://example.com/person/1> :name "John" .
<http://example.com/person/2> :name "Jane" .
```

**Characteristics**:
- Have a unique, global identifier (URI)
- Can be referenced from anywhere
- Have a natural ordering (by URI)

**Pagination**:
```typescript
{
  friends: { take: 10 }  // OK without orderBy
}
```

Works because URIs provide a natural, stable ordering.

### Blank Nodes

**Definition**: Anonymous nodes without URIs.

```turtle
<http://example.com/person/1> :address _:b1 .
_:b1 :city "New York" .
_:b1 :zip "10001" .
```

**Characteristics**:
- No global identifier
- Only accessible through their relationships
- **No natural ordering**

**Pagination Problem**:
```typescript
{
  addresses: { take: 10 }  // DANGEROUS without orderBy!
}
```

❌ **Without orderBy**: Results may vary between queries!

```sparql
SELECT ?address WHERE {
  ?person :address ?address .
}
LIMIT 10
```

Running this twice might return different 10 addresses.

✅ **With orderBy**: Consistent results!

```typescript
{
  addresses: {
    take: 10,
    orderBy: { city: 'asc' }  // Required for blank nodes!
  }
}
```

```sparql
SELECT ?address WHERE {
  ?person :address ?address .
  ?address :city ?city .
}
ORDER BY ASC(?city)
LIMIT 10
```

Now you always get the same 10 addresses (by city name).

## Double-Pagination Prevention

Pagination can happen at two stages in the pipeline:

- **Query Stage**: LIMIT/OFFSET in SPARQL query
- **Extraction Stage**: Array slicing in JavaScript

### The Problem

If we paginate at both stages, we get incorrect results:

<Mermaid chart={`graph TD
    A[100 items in database] -->|SPARQL LIMIT 10| B[10 items in dataset]
    B -->|JS slice take: 5| C[5 items returned]
    
    style C fill:#ffdddd
`} />

❌ **Expected**: 10 items  
❌ **Actual**: 5 items (double pagination!)

### The Solution: Source Tracking

The framework tracks where pagination was applied:

```typescript
{
  "x-pagination": {
    take: 10,
    skip: 0,
    orderBy: { name: 'asc' },
    source: "query"  // ← Marked at query stage!
  }
}
```

**Extraction Stage Logic**:
```typescript
if (paginationMeta?.source === "query") {
  // Already paginated - use all items
  return items;
} else if (paginationMeta) {
  // Apply pagination in-memory
  return items.slice(skip, skip + take);
}
```

### When to Use Each Stage

<Mermaid chart={`graph TD
    A{Large Dataset?} -->|Yes| B[Query-Stage Pagination]
    A -->|No| C[Extraction-Stage Pagination]
    
    B --> D[Less data transferred]
    B --> E[Database handles sorting]
    B --> F[More efficient]
    
    C --> G[Single query fetches all]
    C --> H[Flexible filtering]
    C --> I[Better for CRUD updates]
    
    style B fill:#e1ffe1
    style C fill:#fff4e1
`} />

#### Query-Stage Pagination (Recommended for large arrays)

**Advantages**:
- Reduces data transfer from database
- Database can optimize ORDER BY + LIMIT
- Faster for large datasets

**Use when**:
- Array has many items (>100)
- You only need a specific page
- Read-only operations

**Example**:
```typescript
normalizeSchema(schema, {
  include: {
    posts: { take: 20, orderBy: { date: 'desc' } }
  }
});
// Generated SPARQL includes LIMIT 20
```

#### Extraction-Stage Pagination

**Advantages**:
- Single query fetches all data
- Easier for CRUD operations (need full object)
- More flexible post-query filtering

**Use when**:
- Array is small (&lt;100 items)
- You need the complete object for updates
- Pagination is dynamic/computed

**Example**:
```typescript
// Query without pagination
const result = normalizedSchema2construct(iri, schema);

// Apply pagination during extraction
extractFromGraph(iri, dataset, schema, {
  include: { posts: { take: 20 } }
});
```

### CRUD Update Consideration

**Important**: For updates, you need the **full object**:

```typescript
// ❌ BAD: Partial object (query-stage pagination)
const person = await fetch({
  include: { friends: { take: 10 } }
});
// Only has 10 friends

await update(person);
// 💥 Deletes the other 90 friends!

// ✅ GOOD: Full object
const person = await fetch({
  include: { friends: true }  // All friends
});

await update(person);
// ✅ All relationships preserved
```

## includeRelationsByDefault

Controls whether relationships are automatically included.

### Default Behavior (true)

```typescript
{
  includeRelationsByDefault: true  // default
}
```

**Result**: All relationships are included unless explicitly excluded.

```typescript
{
  name: "John",
  friends: [/* all friends */],
  posts: [/* all posts */],
  employer: {/* employer data */}
}
```

### Explicit Mode (false)

```typescript
{
  includeRelationsByDefault: false,
  include: {
    friends: true  // Explicitly include
  }
}
```

**Result**: Only explicitly included relationships.

```typescript
{
  name: "John",
  friends: [/* friends */]
  // posts and employer are excluded
}
```

### When to Use Each

| Mode | Use Case |
|------|----------|
| `true` (default) | Fetching complete objects, CRUD operations |
| `false` | Performance optimization, selective loading |

## Schema Normalization Depth

Controls how deep `$ref` references are resolved.

```typescript
{
  maxRecursion: 4  // Default: resolve refs up to 4 levels deep
}
```

### Circular References

For circular schemas like:

```typescript
{
  type: "object",
  properties: {
    name: { type: "string" },
    friends: {
      type: "array",
      items: { $ref: "#" }  // ← Circular: Person → Person
    }
  }
}
```

**Without depth limit**: Infinite expansion!

**With maxRecursion: 2**:
```typescript
Person {
  name: string,
  friends: Person[] {
    name: string,
    friends: Person[] {
      name: string,
      friends: {}  // ← Stopped at depth 2
    }
  }
}
```

### Visual Example

<Mermaid chart={`graph TD
    A[Person] -->|friends| B[Person]
    B -->|friends| C[Person]
    C -->|friends| D[Person]
    D -.maxRecursion reached.-> E[Stub]
    
    style E fill:#ffdddd
`} />

**Best Practice**: Set `maxRecursion` based on your data structure:
- Flat data: one to two levels
- Moderate nesting: three to four levels (default)
- Deep hierarchies: five to six levels
- Very deep: Consider restructuring your schema

## Type Coercion

The extractor automatically coerces RDF literals to JavaScript types.

### String Coercion

```typescript
// RDF: "John"^^xsd:string
// Schema: { type: "string" }
// Result: "John"
```

### Number Coercion

```typescript
// RDF: "42"^^xsd:integer
// Schema: { type: "number" }
// Result: 42

// RDF: "3.14"^^xsd:decimal
// Schema: { type: "number" }
// Result: 3.14
```

### Boolean Coercion

```typescript
// RDF: "true"^^xsd:boolean
// Schema: { type: "boolean" }
// Result: true
```

### Date Coercion

```typescript
// RDF: "2024-01-15"^^xsd:date
// Schema: { type: "string", format: "date" }
// Result: "2024-01-15" (string)

// Or with custom handling:
// Result: new Date("2024-01-15") (Date object)
```

## Injection Prevention

All SPARQL queries are built using `@tpluscode/sparql-builder`, which:

✅ Automatically escapes all values  
✅ Prevents SPARQL injection  
✅ Validates RDF terms  
✅ Handles special characters safely  

**You don't need to do anything** - it's automatic!

### Example

**User Input**:
```typescript
const userInput = '"; DROP GRAPH <http://example.com>; SELECT * WHERE { ?s ?p "'
```

**Without Protection** (❌ vulnerable):
```sparql
SELECT * WHERE {
  ?person :name ""; DROP GRAPH <http://example.com>; SELECT * WHERE { ?s ?p "" .
}
```

**With Query Builder** (✅ safe):
```sparql
SELECT * WHERE {
  ?person :name "\"; DROP GRAPH <http://example.com>; SELECT * WHERE { ?s ?p \"" .
}
```

The input is properly escaped as a literal string value.

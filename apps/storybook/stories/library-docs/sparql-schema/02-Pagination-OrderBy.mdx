# Pagination & ORDER BY

Pagination with consistent ordering is essential for reliable, repeatable results, especially when dealing with blank nodes.

## Prisma-Style Pagination API

The framework uses Prisma's familiar pagination syntax:

```typescript
{
  friends: {
    take: 10,          // LIMIT 10
    skip: 20,          // OFFSET 20
    orderBy: { name: 'asc' }  // ORDER BY ASC(?name)
  }
}
```

## ORDER BY Support

### Single Sort Criterion

```typescript
{
  orderBy: { name: 'asc' }
}
```

**Generated SPARQL**:
```sparql
ORDER BY ASC(?name)
```

### Multiple Sort Criteria

```typescript
{
  orderBy: [
    { age: 'desc' },
    { name: 'asc' }
  ]
}
```

**Generated SPARQL**:
```sparql
ORDER BY DESC(?age) ASC(?name)
```

## Why ORDER BY Matters

### Named Nodes (with IRIs)

Named nodes have natural ordering by IRI:

```turtle
<http://example.com/person/1> :name "John" .
<http://example.com/person/2> :name "Jane" .
```

Even without `ORDER BY`, results are somewhat predictable.

### Blank Nodes (anonymous)

Blank nodes have **no natural ordering**:

```turtle
_:b1 :name "John" .
_:b2 :name "Jane" .
```

Without `ORDER BY`, results may vary between queries!

**Best Practice**: Always use `ORDER BY` with blank node pagination.

## Generated SPARQL

When pagination includes `orderBy`, a SUBSELECT is generated:

**Input**:
```typescript
{
  friends: {
    take: 5,
    orderBy: { name: 'asc' }
  }
}
```

**Output**:
```sparql
WHERE {
  OPTIONAL { {
    SELECT ?friends_0 WHERE {
      <subject> :friends ?friends_0 .
      OPTIONAL { ?friends_0 :name ?name }
    }
    ORDER BY ASC(?name)
    LIMIT 5
  } }
  OPTIONAL { ?friends_0 :name ?name_1 }
}
```

## Pagination Metadata

The generator marks pagination with `source: "query"`:

```typescript
{
  paginationMetadata: Map {
    "friends" => {
      take: 10,
      orderBy: { name: 'asc' },
      source: "query"  // ← Applied at query stage
    }
  }
}
```

This prevents double-pagination during extraction.

## Examples

### Example 1: Simple Pagination

```typescript
normalizeSchema(schema, {
  include: {
    posts: { take: 20 }
  }
});
```

**Generated**:
```sparql
SELECT ?posts_0 WHERE {
  <subject> :posts ?posts_0 .
}
LIMIT 20
```

### Example 2: With Offset

```typescript
{
  posts: {
    take: 20,
    skip: 40
  }
}
```

**Generated**:
```sparql
LIMIT 20
OFFSET 40
```

### Example 3: Full Pagination

```typescript
{
  comments: {
    take: 10,
    skip: 0,
    orderBy: [
      { createdAt: 'desc' },
      { id: 'asc' }
    ]
  }
}
```

**Generated**:
```sparql
ORDER BY DESC(?createdAt) ASC(?id)
LIMIT 10
```

## Performance Considerations

- `ORDER BY` can be expensive on large datasets
- Ensure sort properties are indexed in your triple store
- Use `LIMIT` to reduce result size
- Consider query-stage vs. extraction-stage pagination

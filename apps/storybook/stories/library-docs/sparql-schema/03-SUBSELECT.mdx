# SUBSELECT for Pagination

When pagination includes `take`, `skip`, or `orderBy`, the generator creates a SUBSELECT query for optimal performance.

## What is a SUBSELECT?

A SUBSELECT is a nested SELECT query inside a WHERE clause:

```sparql
WHERE {
  OPTIONAL { {
    SELECT ?item WHERE {
      <subject> :items ?item .
    }
    LIMIT 10
  } }
}
```

## When is SUBSELECT Used?

SUBSELECT is generated when an array property has:
- `take` (LIMIT)
- `skip` (OFFSET)
- `orderBy` (ORDER BY)

**Without pagination**:
```sparql
WHERE {
  OPTIONAL { <subject> :friends ?friends }
}
```

**With pagination**:
```sparql
WHERE {
  OPTIONAL { {
    SELECT ?friends WHERE {
      <subject> :friends ?friends .
    }
    LIMIT 10
  } }
}
```

## Complete SUBSELECT Structure

```sparql
WHERE {
  OPTIONAL { {
    SELECT ?item WHERE {
      <subject> :items ?item .
      OPTIONAL { ?item :sortKey ?sortKey }  # For ORDER BY
    }
    ORDER BY ASC(?sortKey)
    LIMIT 10
    OFFSET 20
  } }
  OPTIONAL { ?item :prop1 ?prop1 }  # Extract item properties
  OPTIONAL { ?item :prop2 ?prop2 }
}
```

## Example: Full Pagination

**Input**:
```typescript
{
  friends: {
    take: 5,
    skip: 10,
    orderBy: { name: 'asc' }
  }
}
```

**Generated SPARQL**:
```sparql
CONSTRUCT {
  <subject> :friends ?friends_0 .
  ?friends_0 :name ?name_1 .
}
WHERE {
  OPTIONAL { {
    SELECT ?friends_0 WHERE {
      <subject> :friends ?friends_0 .
      OPTIONAL { ?friends_0 :name ?name }
    }
    ORDER BY ASC(?name)
    LIMIT 5
    OFFSET 10
  } }
  OPTIONAL { ?friends_0 :name ?name_1 }
}
```

## Why Use SUBSELECT?

### Data Transfer Efficiency

**Without SUBSELECT** (fetch all, paginate later):
```
Database → 1000 friends → Network → Client → Filter to 10
```

**With SUBSELECT** (paginate at source):
```
Database → 10 friends → Network → Client
```

Reduces network traffic by 100x!

### Database Optimization

SPARQL engines can optimize:
- `LIMIT` to stop early
- `ORDER BY` with indexes
- `OFFSET` for efficient skipping

## ORDER BY in SUBSELECT

ORDER BY properties must be fetched in the SUBSELECT:

```sparql
SELECT ?item WHERE {
  <subject> :items ?item .
  OPTIONAL { ?item :sortKey ?sortKey }  # ← Needed for ORDER BY
}
ORDER BY ASC(?sortKey)
```

## Multiple Sort Criteria

```typescript
{
  orderBy: [
    { age: 'desc' },
    { name: 'asc' }
  ]
}
```

**Generated**:
```sparql
SELECT ?item WHERE {
  <subject> :items ?item .
  OPTIONAL { ?item :age ?age }
  OPTIONAL { ?item :name ?name }
}
ORDER BY DESC(?age) ASC(?name)
```

## Blank Nodes & ORDER BY

For blank nodes, ORDER BY is **required** for consistent pagination:

```typescript
{
  addresses: {  // Blank nodes
    take: 10,
    orderBy: { city: 'asc' }  // ← Required!
  }
}
```

Without ORDER BY, blank node order is undefined and may vary between queries.

## Implementation

The generator uses `@tpluscode/sparql-builder` for safe SUBSELECT construction:

```typescript
import { SELECT } from "@tpluscode/sparql-builder";

let query = SELECT`${objectVar}`
  .WHERE`${subject} ${predicate} ${objectVar}`;

if (orderBy) {
  // Add ORDER BY properties to WHERE
  query = query.WHERE`OPTIONAL { ${objectVar} ${sortProp} ${sortVar} }`;
}

if (orderBy) {
  query = query.ORDER().BY(sortVar, isDesc);
}

if (take) {
  query = query.LIMIT(take);
}

if (skip) {
  query = query.OFFSET(skip);
}
```

## Performance Tips

- Use ORDER BY on indexed properties
- Keep LIMIT small for faster queries
- Consider caching for frequently accessed pages
- Monitor triple store query performance
